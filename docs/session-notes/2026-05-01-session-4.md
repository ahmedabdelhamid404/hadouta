## Bugs / blockers + how we got past them

1. **Vercel MCP OAuth invalid redirect URL.** The Anthropic-provided OAuth client config and Vercel's redirect-URI allowlist were out of sync. Skipped MCP, used the Vercel CLI direct.

2. **`gh auth login --web` device-code display issue.** Ahmed couldn't see the one-time code printed to terminal — output may have been outside his scrollback or buffered. Eventually got through it manually. Lesson: when the device-code flow fails twice for a user, switch to fine-grained PAT immediately (next session, if relevant).

3. **GitHub `workflow` scope rejection on push.** OAuth token from `gh auth login` had `repo` but not `workflow` — push rejected for modifying `.github/workflows/ci.yml`. Two attempts at `gh auth refresh -h github.com -s workflow` didn't actually update scopes (status still showed old set). **Workaround**: switched both repos' remotes from HTTPS to SSH (`git remote set-url origin git@github.com:...`). SSH push uses the user's identity directly, has no OAuth-app-level workflow restriction. Pushes worked first try.

4. **Railway `--variables` flag echoes secrets to stdout.** Tried to set env vars in one shot via `railway add --service X --variables "K=V"`. Railway's CLI echoes each `--variables` value back as a confirmation prompt, which leaked the freshly-rotated `DATABASE_URL` and the `BETTER_AUTH_SECRET` to the chat transcript. Mitigation: rotated both credentials again (Neon password + BETTER_AUTH_SECRET regenerated), then split the workflow:
   - Service creation: `railway add --service hadouta-backend` (no vars)
   - Env vars: `railway variable set --skip-deploys` for non-secrets; `railway variable set --stdin KEY` (with value piped from a 0600 temp file) for secrets
   - Final deploy: `railway up --service hadouta-backend --detach`

5. **`railway add --repo` returned `Unauthorized`** while `railway whoami` worked. Cause: the GitHub-Railway integration needs a separate Railway-side authorization on the user's GitHub. We dropped `--repo` and went CLI-driven for now. **Followup**: set up Railway GitHub integration in the dashboard for push-to-deploy parity with Vercel.

6. **Vercel deploy-id URLs walled by 401.** Disabled `ssoProtection` via REST API. The bare alias was always public; only the deploy-id and team-namespaced URLs were walled.

## Notes / observations

- **Three CLIs in one session, one common pattern.** Each platform (Vercel, Railway, Neon) split auth from action: interactive browser login once, then everything else is non-interactive script-driven. This pattern compounds — every future deploy iteration is faster because the wiring is in place. Worth treating as the modern default rather than dashboard-clicking.
- **CLI `--variables` echoes are an industry-wide hygiene gap.** Both Railway and (less severely) Vercel echo command-line `--var KEY=VALUE` flags as confirmation. The right pattern for secrets is *always* stdin: `cat secret.txt | tool set --stdin KEY`. Worth codifying as a project rule going forward — "secret-bearing flags must use stdin." Adding to skills-roadmap candidates.
- **Vercel's `bare-alias is public, deploy-id URL is walled` design** is non-obvious and confused us for a few minutes. The takeaway: production traffic should always go to the bare project alias (`hadouta-web.vercel.app`), never the deploy-id URL. When you later add a custom domain (`hadouta.com`), it points to the same bare alias.
- **Manager pattern lite this session.** No specialist agent delegations — this was direct manager work because the unit of progress was "shell command + verify" rather than "design + code + review." Specialist delegation makes sense for code where multiple competent reviewers add value; for sequential infra steps, the bottleneck is the chain itself, not the judgment per step.
- **The `--skip-deploys` flag on `railway variable set` is a quietly important UX choice.** Setting 6 env vars without it would have triggered 6 redeploys back-to-back. With it, all changes batch into one explicit `railway up`. Other CLIs should adopt this pattern. (Vercel doesn't need it — env vars are decoupled from deploys, requiring an explicit redeploy to take effect.)
- **The chained-leak pattern.** This session leaked credentials to the transcript twice (Vercel `vca_` token via `cat`, Railway env vars via CLI echo). Each was caught and rotated, but it cost ~10 minutes total. Codifying: any time I'm reading a file or passing a secret as an argument, the default should be `head -c 0` / `--stdin` / `>/dev/null`, with full-content reads requiring explicit justification.
