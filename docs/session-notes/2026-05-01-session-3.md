### Bugs caught + fixed in-flight

1. **Drizzle-kit emitted broken migration SQL.** Line 53 was `ALTER TABLE "orders" DROP CONSTRAINT "orders_user_id_users_id_fk"`, but line 52's `DROP TABLE "users" CASCADE` had already removed it as a cascade side-effect. First `pnpm db:migrate` failed with "constraint does not exist". Fix: added `IF EXISTS`. Drizzle wraps each migration in a transaction by default → the first attempt rolled back atomically and dev DB was unharmed. Verified via direct introspection of `pg_constraint` and `information_schema.tables` before retrying.

2. **`.min(1).optional()` Zod tightening broke the dev path.** Code Reviewer's should-fix #2 suggested replacing `z.string().optional()` with `z.string().min(1).optional()` for stronger empty-string semantics. Senior Developer applied it. But `.optional()` only allows `undefined`, not empty string — so `.env`'s `GOOGLE_CLIENT_ID=` placeholders fell through to `.min(1)` and rejected validation, blocking startup in dev. Fix: introduced an `optionalNonEmpty` helper using `z.preprocess((v) => v === "" ? undefined : v, z.string().optional())`. The Code Reviewer's brief had actually mentioned this exact alternative; it just wasn't the path the implementer chose.

## Notes / observations

- **Manager pattern paid off.** The parallel Security + Code review caught one blocker and 7 actionable should-fixes the implementer alone wouldn't have. Cost: ~3 agents in parallel ≈ 1 single-agent review pass token-wise. Worth the spend for auth code.
- **One reviewer's "tighten" was another bug.** Code Reviewer's `.min(1).optional()` should-fix solved a theoretical problem but broke the documented `.env` placeholder pattern. Lesson: when a reviewer's correction doesn't have a paired test demonstrating the bug, treat it as a hypothesis. Saved by `pnpm test` catching it on retry.
- **Drizzle-kit's CASCADE-aware migration generation is a real gap.** Two reviewers + an implementer didn't catch the `DROP CONSTRAINT` redundancy because they were reviewing SQL syntax, not Postgres dependency semantics. Worth a `superpowers:` skill candidate: "validate-drizzle-migration" — a pre-flight check that runs the generated SQL against a scratch DB before committing.
- **The lazy-getter circular-import workaround in `email.ts`** is a real-world artifact of "single Zod env schema" + "transport module that needs config from it." A 5-line workaround documented at line 11-23. Acceptable; not a smell.
- **Token efficiency.** This session ran 3 spawned agents (~190K total tokens) plus manager direct work. Comparable to session 2's "~70% complete with no specialist delegation" but delivered a much harder feature. The manager pattern is paying for itself once tasks cross the senior-judgment threshold.
- **Skill candidate noticed** — `validate-drizzle-migration`: pre-flight a generated migration SQL against a scratch DB before committing. Would have caught both the CASCADE-redundant-DROP and any future similar bug. Add to `docs/skills-roadmap.md` if not already there.
