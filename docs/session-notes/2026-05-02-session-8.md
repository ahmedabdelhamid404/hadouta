### 8. Phase 5 Task 1.10 — Paymob payments backend

`hadouta-backend/src/lib/paymob.ts`:
- `createIntention(input)` — POSTs `/v1/intention/` with `Authorization: Token ${SECRET_KEY}`, includes amount + payment_methods + items + billing_data + extras + per-intention `redirection_url` + `notification_url`
- Returns `{ clientSecret, paymobIntentionId, checkoutUrl }` — checkoutUrl built from `${BASE_URL}/unifiedcheckout/?publicKey=${PUBLIC_KEY}&clientSecret=${client_secret}`
- `verifyWebhookHmac(obj, hmacFromQuery)` — SHA-512 over Unified Checkout's 20-field canonical concatenation, constant-time comparison via `crypto.timingSafeEqual`
- `isPaymobConfigured()` — graceful-degradation check

`hadouta-backend/src/routes/payments.ts`:
- `POST /api/payments/intent` — verifies order is in `pending_payment`, creates intention, persists `paymobOrderId` + `paymentId` (clientSecret) on order, returns `{ checkoutUrl }`
- `POST /api/payments/webhook` — Paymob server-to-server callback. HMAC verification logs failures but doesn't reject in dev (Sprint 2 hardening). Maps Paymob outcome → status: `paid` / `pending_payment` / `failed`.
- `GET /api/payments/return` — browser redirect target. Reads `order_id` from query (we passed via `redirection_url`), reads `success` from Paymob's appended params. Optimistic update (marks `paid` so confirmation renders instantly; webhook reconciles). Redirects to `/wizard/7?orderId=…` on success or `/wizard/6?error=payment_failed&reason=…` on failure.

## Notes / observations

- **The verify+sync pattern across Cloudinary + Paymob + Twilio is the same shape three times.** Pattern: umbrella `.env.local` is source of truth → `scripts/<service>/verify-creds.sh` validates → `scripts/<service>/sync-to-railway.sh` pushes to prod env. By session 9+ this pattern can become a templated skill — or a single `scripts/sync-credentials.sh` that handles all services at once.
- **The Cloudinary key swap was a real-world UX issue**: Cloudinary's dashboard labels API Key + API Secret next to each other and they look superficially similar (both are alphanumeric). Verify script caught it within 10 seconds. Without the verify script, this would have been a debug-mode session ("photo upload failing — why?"). Investing in verify scripts is a force-multiplier.
- **Pivot to Cloudinary instead of R2 paid off twice**: no credit card AND auto image transformations save Phase 5 Task 1.9 backend code. About 30 minutes of code total vs 1 hour for R2.
- **The Unified Checkout vs Accept API distinction matters for Paymob** — and Paymob's docs are spotty about which keys go with which API. The `egy_sk_test_*` / `egy_pk_test_*` format is the single best signal that you're on Unified Checkout. Documented in `.env.local` so future Ahmed (or future Claude) doesn't wonder why the legacy Accept API examples don't match the keys.
- **Dev-mode OTP bypass is more elegant than Twilio sandbox for now**. Sandbox requires per-phone opt-in via "join code" (friction for testing), trial accounts can only message verified phones (more friction), and it's a temporary state until Meta Verification anyway. Hardcoded `123456` skips all that for the cost of one constant + one error-message tweak.
- **The Paymob "two callback URLs" question caught me off-guard initially** — I'd written only the webhook (server-to-server) callback. The redirect (browser) callback is essential for UX. Without it, customers' browsers land on a blank Paymob "Done!" page after payment instead of bouncing back to step 7. Always implement both.
- **Optimistic update in `/api/payments/return` is the standard Stripe-style pattern.** Mark order paid in the redirect (since success param tells us so), then let webhook reconcile. Means customer sees "thanks for paying" instantly instead of staring at a 1-3-second spinner waiting for webhook. Same correctness guarantees (webhook is source of truth), much better UX.
